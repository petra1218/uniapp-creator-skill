# 最佳实践设计指南

## 目的

用于在设计初期快速梳理产品骨架，区分「可直接采用的最佳实践」与「需要用户确认的决策点」，避免把设计讨论变成无结构的线性追问。

## 使用时机

只要产品属于通用型品类（转写工具、待办、笔记、商城、会员订阅、内容发布、文件管理等），在进入细节设计之前先完成本指南，再输出设计结论。

## 固定流程

### 第一步：完成产品框定面板

在讨论具体功能之前，先用框定面板收口：

| 维度 | 必须回答 |
|------|---------|
| **目标用户** | 谁用？典型用户画像是什么？ |
| **核心场景** | 用户在什么时间、地点、设备下完成什么任务？ |
| **核心闭环** | 从进入产品到完成目标的最小路径是什么？ |
| **关键资源** | 产品里最重要的数据实体是什么？生命周期是什么？ |
| **价值交付** | 用户离开时带走的成果是什么？ |
| **商业规则** | 权益、计费、会员、邀请/分成等商业逻辑是什么？ |
| **平台边界** | 最终要上线到哪些平台？哪些能力是平台强制的？ |
| **运营与风控** | 需要后台管理的能力有哪些？高危操作边界在哪里？ |

### 第二步：搜索同类产品最佳实践

对标市场上已有的成熟产品，梳理：

- **常见稳定模式**：该品类被市场验证过的核心交互模式
- **默认采用项**：无特殊情况直接采用，不反复讨论
- **平衡型备选**：存在 2-3 种合理方案，但需要给出一个推荐
- **主动不采用项**：有充分理由不采用某模式，明确写出原因
- **真正待确认的问题**：只有影响核心实现路径的问题才保留

## 常见品类的快速默认结论

以下品类已有成熟默认骨架，设计初期可直接引用，再按项目实际情况调整：

### 转写 / 语音处理类产品

**默认采用：**
- 客户端直传云存储，不走云函数中转
- 任务状态用 `pending/submitting/processing/completed/failed` 五态机
- 轮询进度（5s间隔，最大60次）+ 回调双保险
- 配额扣减在云对象层做原子操作
- 免费赠送在用户首次创建任务前完成检查

**平衡型备选：**
- 实时转写 vs 异步转写 → 推荐异步（百度 ASR 回调模式）
- 前端直调百度 vs 云函数代理 → 推荐云函数代理（保护密钥）

**不主动采用：**
- 客户端直接拿百度 ASR 的 access_token（密钥暴露风险）
- 任务状态只用前端本地存储（刷新即丢失）

### 会员订阅类产品

**默认采用：**
- `uni-pay` 作为支付聚合层
- 会员权益用独立 `rc_entitlements` 表，按 `term_id` 关联
- 权益生效时间精确到秒，到期自动失效
- 邀请奖励用独立 `rc_invite_rewards` + `rc_referrals` 表
- 管理员放行用 `rc_audit_log` 记录 before/after snapshot

**平衡型备选：**
- 叠加购买 vs 替换购买 → 推荐叠加（不改变已购有效期）
- 邀请码手动输入 vs 分享链接 → 推荐分享链接（转化更高）

**不主动采用：**
- 余额字段直接扣减（用 `rc_quota_ledger` 流水表更安全）
- 权益用布尔字段（无法追踪来源和历史）

### 管理后台类产品

**默认采用：**
- `uni-admin` 作为基线框架
- 菜单用动态菜单（`opendb-admin-menus`），不硬编码静态菜单
- 数据权限通过 DB Schema `permission` 表达式控制
- 高危操作（放行、导出、删除）需二次确认 + 写审计日志
- 金额、手机号等敏感信息在展示层脱敏

**不主动采用：**
- 后台与客户端混用同一个项目（入口必须隔离）
- 管理员用普通用户角色（`admin` 角色与普通 `role` 分离）

## 设计输出要求

- `docs/design/best-practices.md` 必须在进入详细设计之前完成
- 「默认采用项」不要逐项拿来问用户偏好
- 「真正待确认的问题」不超过 5 条（超过说明设计还没收口）
- 每次设计讨论后回补本文件，不把最佳实践结论只留在对话里
