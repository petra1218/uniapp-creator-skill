# 平台差异核对清单

## 先分类

所有差异先归类到以下三种之一：

1. 编译期差异
2. 运行期差异
3. 能力差异与降级

在分类前，先写清：
- 哪些页面、流程、权限边界、数据结构应保持跨端统一
- 哪些差异只是平台实现差异，不应上升为业务角色差异

不要先问“微信小程序/H5/App/Harmony分别承担什么角色”，而要先问“统一主流程是什么，哪些能力因为平台限制需要降级”。

uni-app 的默认技术目标是一套代码多端输出。最终发行到哪个平台由 HBuilderX/CLI 在发行阶段选择；设计和开发阶段只记录已经明确的发行端要求、技术隔离点和需要实测的平台能力，不把发行端选择当成产品分叉。

## 编译期差异

检查：
- 是否必须使用条件编译
- 是否只是样式、组件、目录差异
- 是否可以保持主流程共享

适合：
- 平台专用入口
- 平台专用组件片段
- 条件编译包裹的小段差异

## 运行期差异

检查：
- 生命周期、导航、容器行为是否一致
- Web 横屏与移动竖屏交互是否一致
- 页面性能或滚动行为是否需要单独处理

适合：
- 运行时分支
- 差异化交互补丁

## 能力差异与降级

检查：
- 是否存在 API、支付、广告、登录、系统能力差异
- 是否可以降级、替代或隐藏
- 用户是否真的需要全平台完全一致

处理原则：
- 先保统一主流程
- 再做能力裁剪
- 最后才考虑平台专用补丁或平台专用页面
- 除非产品明确要求，否则不要把某端预设为“轻量端”“传播端”并反向裁掉主业务

默认优先级：
1. 降级
2. 替代
3. 隐藏
4. 平台专用实现

## 平台提醒

- 主动向用户询问发行端时，只在存在上线计划、平台能力风险、合规或成本差异时询问
- 未明确发行端时，默认按 uni-app 多端兼容主线设计和实现，不阻塞概要设计
- `Harmony`：出现明确发行或能力差异时单独记录技术隔离点
- 小程序：默认接受能力不完全对齐
- H5/Web：默认接受“Web 增强”，不强求具备移动端全部能力
- App：优先保持主流程共享，不提前拆成 iOS/Android 两套
- 微信小程序：默认先按统一主流程承接业务，再根据官方能力限制做条件编译、替代或隐藏

## H5 / SSR / 分发提醒

不要直接把 H5/Web 定义成“轻端”。只有产品存在公开 Web 页面、SEO、营销落地页、可分享内容页，或用户明确要求 H5 发布时，才进一步判断：
- 是否是登录后工具型页面，还是公开内容/营销/SEO 页面
- 是否需要 SSR
- 是否需要前端网页托管
- 是否需要 `uni-ssr` 云函数
- Web、SSR 云函数、业务云函数是否同服务空间
- 如果不同服务空间，跨域、cookie、token、请求域名如何处理

H5 的差异优先归类为发布方式、浏览器能力、支付承接、文件能力和 SEO/SSR 需求，不自动上升为业务角色差异。
