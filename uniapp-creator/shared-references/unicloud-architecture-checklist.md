# uniCloud 架构核对清单

## 使用方式

凡是方案或实现涉及 `uniCloud`，先按本清单确定官方主路径，再决定是否需要自定义实现。

不要把 `uniCloud` 简化成“云函数后端”。优先按官方体系判断：`uni-id`、`DB Schema`、`clientDB`、云对象、云函数、URL 化、云存储、前端网页托管、SSR。

## 1. 数据访问路径

先判断每类数据读写适合哪条路径：

1. `clientDB + DB Schema + uni-id`
   - 适合权限边界清晰的列表、详情、个人数据、只读结果、简单增删改查
   - 必须写清 `DB Schema` 权限表达式、字段校验、索引与数据归属字段
2. 云对象
   - 适合需要业务编排、事务式校验、额度扣减、状态推进、组合查询的应用逻辑
3. 云函数
   - 适合定时任务、后台任务、第三方回调、URL 化 HTTP 入口、复杂服务端处理
4. 自定义接口或第三方直连
   - 只有官方路径不适合，或第三方明确要求时才使用

设计输出里不能只写"云端处理"。必须明确每个模块使用哪条路径，以及为什么。

### 1.1 DB Schema 格式强制（跨文档统一规则）

**所有数据模型定义必须使用 uniCloud 官方 DB Schema 格式**，可直接部署为 `uniCloud/database/{collection-name}.schema.json`。

**禁止格式：** `columns: [...]` 数组格式

**必须格式：**

```json
{
  "bsonType": "object",
  "required": ["field1", "field2"],
  "permission": {
    "read": "权限表达式",
    "create": "权限表达式",
    "update": "权限表达式",
    "delete": "权限表达式"
  },
  "properties": {
    "field1": {
      "bsonType": "string",
      "label": "字段说明",
      "description": "详细描述"
    },
    "field2": {
      "bsonType": "number",
      "label": "字段说明"
    }
  }
}
```

**最低要求：**
- `bsonType`: 必须为 `"object"`
- `required`: 列出所有必填字段
- `permission`: 必须包含 `read`、`create`、`update`、`delete` 四种操作的权限表达式
- `properties`: 必须使用对象格式（`{}`），每个字段必须包含 `bsonType` 和 `label`
- `indexes`: 高频查询字段必须列出索引定义

**跨文档一致性规则：**
- `data-model.md` 中定义的每个集合，在 `api-contract.md` 中必须有对应的云对象/云函数方法
- `api-contract.md` 中每个云对象/云函数方法操作的集合，必须在 `data-model.md` 中有定义
- 如果发现不一致，以 `data-model.md` 为基准补全缺失方

## 2. uni-id 与权限边界

只要写“共享 `uni-id` 用户体系”，必须继续写清：

- 用户端是否使用 `uni-id-pages`
- 云端是否使用 `uni-id-common` 校验登录态
- 哪些集合通过 `DB Schema` 约束 `auth.uid`
- 后台是否基于 `uni-admin` / 角色权限控制
- 多应用共享服务空间时，哪些集合必须带 `app_id`、租户字段、业务归属字段
- 哪些授权不能只依赖 `dcloudAppid`

权限设计至少覆盖：用户本人、后台管理员、系统任务、第三方回调。

## 3. 云存储路径

出现文件、图片、音频、视频、附件时，必须拆成四个设计面：

1. 上传
   - 一般优先客户端直传 `uniCloud` 云存储
   - 不让云函数默认中转大文件
2. 访问
   - 明确公有读、私有读、临时链接、后台人工访问、第三方访问
   - 私有文件优先由云端生成临时链接
3. 删除
   - 删除优先由云端统一鉴权执行
   - 不把前端删除当默认实现
4. 生命周期
   - 原文件保留多久
   - 结果文件保留多久
   - 失败任务文件如何清理
   - 用户注销、任务关闭、人工处理完成后如何处理

数据模型至少要能记录：`file_id`、存储路径、可见性、临时链接用途、过期策略、删除状态、删除操作者、删除时间。

## 4. 外部系统与 URL 化

出现第三方回调、Webhook、外部系统访问 `uniCloud` 时，必须输出 URL 化设计：

- 云函数 URL 化还是云对象 URL 化
- HTTP 路径与方法
- 鉴权方式
- 签名、时间戳、nonce 或等价防重放策略
- 幂等键
- 重试策略
- 原始请求是否落库
- 原始返回如何标准化
- 失败码、重试次数、人工介入条件

流程图里不能只画“第三方 -> Cloud”。必须标清 URL 化入口和回调处理节点。

## 5. 第三方服务接入合同

出现支付、语音转写、OCR、AI、内容安全、消息、地图等第三方能力时，必须输出最小合同：

- provider 名称与能力边界
- provider request id / task id
- 本地任务 id 与 provider id 的映射
- 提交参数摘要
- 回调或轮询方式
- 成功、失败、处理中、超时状态映射
- 失败码与可重试规则
- 幂等与重复回调处理
- 结果标准化字段
- 成本、配额、并发、限流风险

如果第三方要求直接访问文件，必须说明它拿的是临时 URL、云端下载结果，还是客户端本地文件。

## 6. Web、SSR 与分发

只有产品存在公开 Web 页面、SEO、营销落地页、可分享内容页，或用户明确要求 H5/Web 发布时，才需要判断：

- 这是登录后工具型 Web，还是公开内容/营销/SEO 页面
- 是否需要 SSR
- 是否使用前端网页托管
- 如启用 SSR，是否需要 `uni-ssr` 云函数
- Web、SSR 云函数、业务云函数是否在同一服务空间
- 如果不在同一服务空间，跨域、cookie、token、请求域名如何处理

默认不要把 H5/Web 定义成“轻端”。H5/Web 首先是统一主流程的一个可能发行端；SSR/托管只在产品目标需要时作为技术分支记录。

## 7. 状态映射

只要方案出现状态流转，必须输出状态映射表：

| 服务端状态 | 用户端显示状态 | 后台可操作 | 用户可操作 | 是否计费 | 是否可重试 |
|---|---|---|---|---|---|

如果存在后台完整状态与前端简化状态，必须明确映射关系，不允许只列两套状态名称。

## 8. 任务卡闭环

设计里列出的模块编号必须能落到真实任务卡。

每张任务卡至少包含：
- 目标
- 输入文档
- 输出文件
- 实施范围
- 验收标准
- 风险项
- 是否需要 TDD

如果任务卡路径不存在，不能宣称设计已经可交接执行。
