# 官方生态核对清单

## 使用方式
设计或执行时，按顺序过一遍。只要前一层能满足，就不要跳到后一层自定义实现。

## 核对顺序

### 1. 先看项目里是否已经有官方封装
- 是否已有基于 `uni-ui`、`uni_modules`、`uniCloud` 的封装
- 是否已有官方模板遗留的业务骨架
- 是否已有统一登录、权限、云对象调用方式

### 2. 再看官方起点
- `Hello uni-app`
- `uni-ui` 官方组件起点
- `uni-starter`
- `uni-admin`

### 3. 再看官方模块与能力
- `uni-ui`
- `uni_modules`
- `uni-id-pages`
- `uniCloud`
- `schema2code`
- `opendb` / 官方公共数据模型
- `uni-ad`
- 官方 Sketch 设计资源

### 4. 再决定是否最小改造
- 换皮
- 组合
- 补少量业务包装

### 5. 最后才是自定义实现
- 官方能力明显不覆盖
- 官方能力覆盖了，但与当前项目边界冲突且无法通过薄包装解决

## 设计或执行时至少要回答
- 当前方案复用了哪些官方能力
- 哪些地方只是做最小改造
- 哪些地方需要业务包装
- 哪些地方必须自定义，以及为什么
