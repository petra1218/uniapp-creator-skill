# 官方生态核对清单

## 使用方式
设计或执行时，按顺序过一遍。只要前一层能满足，就不要跳到后一层自定义实现。

如果当前项目存在“多个应用共用同一个 `uniCloud` 服务空间”，先不要直接假设官方库都适合共享。必须逐类核查“是否适合进入共享层”和“隔离边界是否成立”。

## 核对顺序

### 1. 先看项目里是否已经有官方封装
- 是否已有基于 `uni-ui`、`uni_modules`、`uniCloud` 的封装
- 是否已有官方模板遗留的业务骨架
- 是否已有统一登录、权限、云对象调用方式

### 2. 再看官方起点
- `Hello uni-app`
- `uni-ui` 官方组件起点
- `uni-starter`
- `uni-admin`

### 3. 再看官方模块与能力
- `uni-ui`
- `uni_modules`
- `uni-id-pages`
- `uniCloud`
- `schema2code`
- `opendb` / 官方公共数据模型
- `uni-ad`
- 官方 Sketch 设计资源

对于多应用共用同一个 `uniCloud` 服务空间，优先评估这些 DCloud 官方推荐、且常见于基础设施层的标准开源库：
- `uni-id`
- `uni-config-center`
- `uni-starter`
- `uni-admin`
- `uni-upgrade-center`
- `uni-pay`
- `uni-cms`
- `uni-im`
- `uni-ad-admin`
- `uSearch`
- `uni-captcha`
- `uni-sec-check`
- `uni-cloud-router`
- `uni-open-bridge`
- `uni-cloud-s2s`
- `uni-subscribemsg`
- `uni-map`

但它们不是默认都能共享，必须继续判断：
- 官方文档是否明确支持多应用共享或按 `dcloudAppid` / 应用维度配置
- 共享后是否会引入认证、授权、支付、第三方凭据、内容安全、消息、存储越权风险
- 是否还需要用户、角色、租户、业务归属字段、路径命名空间等额外隔离
- 是否必须拆成“共享底座 + 应用私有配置”两层

### 4. 再决定是否最小改造
- 换皮
- 组合
- 补少量业务包装

### 5. 最后才是自定义实现
- 官方能力明显不覆盖
- 官方能力覆盖了，但与当前项目边界冲突且无法通过薄包装解决

## 设计或执行时至少要回答
- 当前方案复用了哪些官方能力
- 哪些地方只是做最小改造
- 哪些地方需要业务包装
- 哪些地方必须自定义，以及为什么

如果当前项目是多应用共用同一个 `uniCloud` 服务空间，还至少要回答：
- 哪些官方库进入共享层，理由是什么
- 哪些官方库不进入共享层，理由是什么
- 哪些能力可以用 `dcloudAppid` 做应用级隔离
- 哪些能力不能只靠 `dcloudAppid`，还需要额外授权、配置拆分或资源命名空间
