# 官方生态核对清单

## 使用方式
设计或执行时，按顺序过一遍。只要前一层能满足，就不要跳到后一层自定义实现。

如果当前项目存在“多个应用共用同一个 `uniCloud` 服务空间”，先不要直接假设官方库都适合共享。必须逐类核查“是否适合进入共享层”和“隔离边界是否成立”。

## 核对顺序

### 1. 先看项目里是否已经有官方封装
- 是否已有基于 `uni-ui`、`uni_modules`、`uniCloud` 的封装
- 是否已有官方模板遗留的业务骨架
- 是否已有统一登录、权限、云对象调用方式

### 2. 再看官方起点
- `Hello uni-app`
- `uni-ui` 官方组件起点
- `uni-starter`
- `uni-admin`

### 3. 再看官方模块与能力
- `uni-ui`
- `uni_modules`
- `uni-id-pages`
- `uniCloud`
- `schema2code`
- `opendb` / 官方公共数据模型
- `uni-ad`
- 官方 Sketch 设计资源

`uniCloud` 相关能力必须继续细分，不要笼统写“云端处理”：
- `uni-id` / `uni-id-pages` / `uni-id-common`
- `DB Schema`
- `clientDB`
- 云对象
- 云函数
- 云函数/云对象 URL 化
- 云存储
- 前端网页托管
- `uni-ssr`

默认判断顺序：
1. `clientDB + DB Schema + uni-id` 是否能覆盖读写和权限
2. 是否需要云对象做业务编排
3. 是否需要云函数做后台任务、定时、URL 化或复杂处理
4. 是否涉及云存储上传、临时访问、删除、生命周期
5. H5 是否需要前端网页托管或 SSR
6. 官方路径不适合时，才考虑自定义接口或客户端直连第三方

这是设计者内部使用的判断链，不是每一步都要向用户确认。默认可采用的官方方案应直接进入设计结论；只有存在分支、取舍、成本、合规或风险时才需要拿出来讨论。

如果当前能力涉及文件上传、语音/视频/图片处理、内容审核、附件管理，先把 `uniCloud` 云存储当成官方主路径来核查，而不是先跳到第三方 SDK 接入方式：
- 一般上传文件优先客户端直传 `uniCloud` 云存储
- 云函数更适合做鉴权、排队、回调、结果落库，而不是中转大文件上传
- 私有文件优先通过 `getTempFileURL` 一类云端能力下发临时访问链接
- 删除文件属于高危操作，优先由云函数执行
- 文件生命周期必须明确，包含原文件、结果文件、失败任务文件、用户注销或任务关闭后的清理规则

如果当前能力涉及外部系统、第三方回调或 Webhook，必须优先核查云函数/云对象 URL 化，而不是只写“第三方回调云端”。

对于多应用共用同一个 `uniCloud` 服务空间，优先评估这些 DCloud 官方推荐、且常见于基础设施层的标准开源库：
- `uni-id`
- `uni-config-center`
- `uni-starter`
- `uni-admin`
- `uni-upgrade-center`
- `uni-pay`
- `uni-cms`
- `uni-im`
- `uni-ad-admin`
- `uSearch`
- `uni-captcha`
- `uni-sec-check`
- `uni-cloud-router`
- `uni-open-bridge`
- `uni-cloud-s2s`
- `uni-subscribemsg`
- `uni-map`

但它们不是默认都能共享，必须继续判断：
- 官方文档是否明确支持多应用共享或按 `dcloudAppid` / 应用维度配置
- 共享后是否会引入认证、授权、支付、第三方凭据、内容安全、消息、存储越权风险
- 是否还需要用户、角色、租户、业务归属字段、路径命名空间等额外隔离
- 是否必须拆成“共享底座 + 应用私有配置”两层

### 4. 再决定是否最小改造
- 换皮
- 组合
- 补少量业务包装

### 5. 最后才是自定义实现
- 官方能力明显不覆盖
- 官方能力覆盖了，但与当前项目边界冲突且无法通过薄包装解决

## 设计或执行时至少要回答
- 当前方案复用了哪些官方能力
- 哪些地方只是做最小改造
- 哪些地方需要业务包装
- 哪些地方必须自定义，以及为什么

如果当前项目是多应用共用同一个 `uniCloud` 服务空间，还至少要回答：
- 哪些官方库进入共享层，理由是什么
- 哪些官方库不进入共享层，理由是什么
- 哪些能力可以用 `dcloudAppid` 做应用级隔离
- 哪些能力不能只靠 `dcloudAppid`，还需要额外授权、配置拆分或资源命名空间
- 文件上传、访问链接、删除权限、回调处理分别落在共享层还是应用层
