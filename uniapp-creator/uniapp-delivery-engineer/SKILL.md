# uniapp-delivery-engineer

## 文档信息

- 当前版本：`v1.21`
- 更新时间：`2026-05-27`
- 当前定位：面向执行阶段的官方优先交付

## 概览

用于在既有设计边界内完成 uni-app/uniCloud 开发执行。重点不是“尽快写代码”，而是按当前步骤推进、优先复用官方能力、处理必要的平台差异，并保持文档可持续交接。

这份说明面向真正的开发落地：前端页面、uniCloud、后台协同、文档回写，都要在同一轮里闭环。

默认前提是：uni-app 项目先落实一套代码、一套产品主线、一套共享实现，再用条件编译、运行期判断、能力降级去处理必要的技术隔离点。最终输出哪个平台包由 HBuilderX/CLI 发行阶段决定，开发阶段不要把发行端选择反推成多套业务实现。

执行阶段同样必须保持批判性。不要因为用户、上游设计或已有口头结论给了一个方向，就默认当前实现路径已经足够稳。

至少做到：
- 主动检查当前实现是否跳过了官方路径复核
- 主动检查共享方案是否真的具备隔离边界
- 对高风险能力先做反向核验，再继续落地
- 当设计边界不够清楚时，先回退补设计，而不是顺手打补丁把风险埋下去

不要把批判性变成反复确认。对已经属于 uni-app/uniCloud 官方默认路径的实现，直接按设计和任务卡推进；只有实现会偏离设计、引入技术分支、跨越模块边界或触发成本/权限/平台风险时，才暂停回退补设计。

## 使用前提

只有在下面条件成立时才使用：
- 已经有明确的设计边界
- 已经知道本轮要改哪个模块编号、仓库或目录
- 已经明确官方模板基线或当前项目基线
- 已经能找到对应任务卡或实际执行步骤

如果产品目标、页面流程、模块边界、数据定义或关键技术隔离点仍不清楚，先退回 `uniapp-solution-architect`。

## 开始前先读什么

改代码前按这个顺序读：

1. 先确认任务卡路径是否已经在设计输出中写明
2. 如果还没有已批准的实现计划，先使用 `superpowers:writing-plans` 生成 `docs/superpowers/plans/YYYY-MM-DD-<topic>.md`
3. 如果任务卡路径不明确，先读 `docs/project-system.md`
4. 当前任务卡或当前步骤
5. 相关设计文档
6. 如果存在 `docs/design/best-practices.md`，先读它，确认当前实现没有偏离已收敛的默认模式与主动取舍
7. 如涉及数据，读 `docs/design/data-model.md`
8. 如涉及流程、状态、调用链、平台分支，读 `docs/design/flows.md`
9. 如涉及界面，读 `docs/design/ui-guidelines.md` 中的全局与页面级提示词

不要只凭之前对话记忆直接开工。

如果你发现某项关键设计只存在于 `docs/superpowers/`，而最终 `docs/` 中没有对应落点，先补最终文档，再继续执行。不要把过程稿当成长期事实来源。

如果这些设计文档是由旧版本 skill 产出的，形状还停留在旧骨架，例如：
- `ui-guidelines.md` 只有风格建议，没有可直接复用的页面提示词
- `api-contract.md` 只有统一返回结构，没有 URL 化入口合同
- `overview.md` 只有产品概述，没有建项方式、模板理由、风险项
- 任务卡只有边界说明，没有依赖、执行方式、验证方式

先回补最终 `docs/` 到当前骨架，再继续实施。

## 默认执行顺序

按下面顺序做：

1. 确认本轮只覆盖哪个模块编号和步骤
2. 确认涉及哪个仓库、目录和模块
3. 检查当前项目已有的官方封装和官方模块
4. 选择最小可行改动
5. 完成实现
6. 回写任务卡、总控、变更记录，以及必要的数据文档和流程文档

如果用户一次提了很多事，先把当前模块和当前步骤收窄，不要一轮混做整项目。

## 与常用执行方法的自然配合

执行阶段默认不跳过 superpowers 的计划链路：
- 当前需求还没有实现计划：先用 `superpowers:writing-plans`
- 已经有实现计划，需要按计划逐步落地：显式使用 `superpowers:executing-plans` 或 `superpowers:subagent-driven-development`
- 不要跳过计划文档直接裸实现
- 如果实现要偏离 `docs/design/best-practices.md` 中已经确定的默认模式，先暂停并回退到 `uniapp-solution-architect`

如果当前任务卡只有边界说明，还不够支持直接实施，例如缺少目标目录、依赖关系、执行方式、验证方式或关键页面/路由清单，先用 `superpowers:writing-plans` 把它展开成可执行计划，再动代码。

如果当前任务涉及页面实现，而 `docs/design/ui-guidelines.md` 里没有对应页面的可直接复用提示词，先回退补设计或补文档，不要自己临时编一套视觉口径。

如果 `ui-guidelines.md` 里只有零散风格要点，没有可整段复制的页面 prompt，也视为提示词缺失。

## 执行前文档形状检查

正式实施前，至少快速检查：
- `overview.md` 是否具备建项方式、模板理由、边界、风险项
- `overview.md` 是否只包含一套目录结构（如果出现两套矛盾目录描述，先回退补设计）
- `best-practices.md` 是否具备真正待确认的问题
- `api-contract.md` 是否具备 URL 化入口合同
- `api-contract.md` 中所有云对象/云函数方法对应的数据库集合是否在 `data-model.md` 中存在（如果缺失，先补数据模型）
- `flows.md` 是否具备异常/异步分支
- `data-model.md` 是否使用 uniCloud 官方 DB Schema 格式（`properties: {}` 对象格式，不是 `columns: []` 数组格式）。如果使用了非官方格式，执行前必须修正
- `ui-guidelines.md` 是否具备页面级提示词
- 任务卡是否具备目标目录、依赖、执行方式、验证方式
- 任务卡的目标目录名是否与 `overview.md` 建项结论中的项目名完全一致
- `docs/project-system.md` 中的文档状态是否与实际文件存在情况一致（如果不一致，先更新 project-system.md）
- `docs/tracking/change-log.md` 中各文档的状态标注是否与实际文件存在情况一致（如果实际已生成但标注为"待生成"，先更新 change-log.md）
- `api-contract.md` 是否覆盖所有在任务卡或 `data-model.md` 映射表中引用的方法（包括定时任务触发的云函数；如果缺失，先补 api-contract.md 或回退到 `uniapp-solution-architect`）
- `data-model.md` 中数值型字段的字段名、label、description 是否单位语义一致（如发现字段名含 `minutes` 但 label 写"秒"，或 label 写"价格（分）"但需求使用"元"，先修正 data-model.md）
- **`data-model.md` 中同类型数值字段是否使用统一存储单位。** 全文搜索所有时长字段（label 含"时长""额度""分钟""秒"）、所有金额字段（label 含"金额""价格""元""分"），逐字段核对存储单位是否跨集合统一。若发现不同集合的时长字段分别用"秒"和"分钟"，先修正 data-model.md 统一为一种单位再继续实施
- **确认 DB Schema 的实际部署项目。** 默认放在客户端项目的 `{uniCloud目录名}/database/` 下。如果 `overview.md` 的目录结构中后台项目也出现了 `database/` 目录，在实施前与设计确认实际部署位置并修正 overview.md
- **确认 uniCloud 目录名与目标服务商类型一致。** 从 `overview.md` `## HBuilderX 建项方式` 中提取服务商类型和 uniCloud 目录名，确认两者匹配（阿里云→`uniCloud-aliyun/`、支付宝云→`uniCloud-alipay/`、腾讯云→`uniCloud-tcb/`）。如果 overview.md 未标注服务商类型或目录名，视为设计缺失，先回退到 `uniapp-solution-architect` 补充。**执行时所有云函数、数据库 schema 文件必须写入正确的目录名下**，禁止在未确认服务商的情况下默认写入任何 `uniCloud*` 目录
- **确认跨云对象调用方式已在设计中明确。** 如果设计涉及多个云对象之间的调用（如 `transcription-co` 调用 `quota-co.consumeQuota`），必须确认 `api-contract.md` 中有 `## 跨云对象调用约定` 章节并覆盖所有调用关系。如果缺失，先回退到 `uniapp-solution-architect` 补充设计
- **确认所有任务卡的页面路径与 overview.md 页面清单一致。** 提取 overview.md `## 页面结构与关键入口` 中所有页面路径，逐条搜索每张任务卡的 `## 输出文件` 和 `## 范围`，确认页面路径逐字符匹配。若发现不一致（如 overview 写 `pages/mine/index` 而任务卡写 `pages/profile/index`），以 overview.md 为基准修正任务卡后再继续实施
- **如存在 `docs/design/cross-review-fix-log.md`，先阅读它。** 该文件由 `uniapp-design-reviewer` skill 在中间审查阶段生成，记录了已修复的问题和仍存在的风险项。如果该文件存在，至少确认其中的 P0 问题已全部修复。如果不存在该文件，建议先运行 `uniapp-design-reviewer` 对设计文档进行中间审查再进入实施

缺任一项时，先补文档骨架或回退到 `uniapp-solution-architect`，不要带着半成品设计直接进入实现。

执行时按任务类型自然选择：

- 结构或模块拆分不清：先 `superpowers:writing-plans`
- 已有清晰步骤，进入实际落地：再 `superpowers:executing-plans`
- 行为规则、状态流转、跨端差异、云端逻辑容易回归：优先 `test-driven-development`

这些方法作为执行手段自然接入即可，不要反过来挤占当前开发主线。

## 官方优先执行规则

固定优先顺序：

1. 项目内已有的官方封装
2. 官方模板、官方组件、官方插件、`uni_modules`、官方云能力
3. 最小改造
4. 薄业务包装
5. 官方路径确实不适合时，再做自定义实现

如果当前任务涉及多个应用共用同一个 `uniCloud` 服务空间，默认先检查当前官方模块是否真的适合进入共享层，再决定是共享、薄包装还是拆成应用私有实现。

需要查表时，优先阅读：
- [模板选择指引](references/template-selection-guide.md)（含 HBuilderX 建项详细步骤）
- [官方生态核对清单](references/official-ecosystem-checklist.md)（含 uni-id/uni-admin/uni-pay 最佳实践）
- [uniCloud 架构核对清单](references/unicloud-architecture-checklist.md)
- [最佳实践设计指南](references/best-practices-guide.md)

执行 `uniCloud` 相关任务前，先核验设计是否已明确当前模块使用哪条官方路径：
- `clientDB + DB Schema + uni-id`
- 云对象
- 云函数
- URL 化入口
- 云存储
- 前端网页托管 / SSR

如果设计没有说明，或者实现需要偏离设计路径，先暂停并补设计，不要直接写代码。

如果当前任务涉及文件上传、云存储、媒体转码、语音识别、OCR、内容审核等链路，默认先核验是否遵循 `uniCloud` 官方存储主路径：
- 上传文件优先客户端直传云存储，不让文件先绕云函数中转
- 云函数优先负责鉴权、任务创建、状态同步、回调处理、结果落库
- 私有文件优先通过云函数获取临时链接
- 删除文件优先走云函数，不把前端删除当默认实现
- 原文件、结果文件、失败任务文件必须有生命周期规则
- 后台人工访问、第三方访问、用户预览必须区分临时链接用途

如果当前实现把文件流量先送进云函数再转存，或让前端直接承担高危删除权限，先停下来复核，不要直接继续开发。

如果当前任务涉及第三方回调、支付回调、Webhook 或外部系统访问，执行前必须确认：
- URL 化入口已经在设计中存在
- 鉴权、签名、防重放、幂等键已经明确
- provider request id / task id 与本地任务 id 的映射字段已经存在
- 失败码、重试次数、重复回调处理和人工介入条件已经明确

缺任一项时，先回退补设计与数据模型。

## 代码生成强制规范

以下规范在**所有代码生成场景**中强制遵守，不得跳过或降级。规范分为：客户端、服务端、平台工程、目录结构、注释五大类。

### 客户端开发规范

#### 框架与语法

- **必须**使用 Vue 3 `<script setup>` 语法，采用 Composition API
- 页面类型默认使用 `.vue`，仅在需要高性能长列表或原生渲染等特定场景下使用 `.nvue`
- **禁止**使用已废弃的 Vue 2 语法（Options API、filters 等）

#### 多端兼容与条件编译

- **必须**兼顾 H5、小程序、App、鸿蒙等多端兼容性
- 遇到平台特有 API 或差异时，**必须**使用 `#ifdef` / `#ifndef` / `#endif` 进行条件编译处理，**禁止**用 JS `if` 逻辑判断平台来替代条件编译
- Android 和 iOS 平台**不支持**通过条件编译区分，需通过 `uni.getSystemInfoSync().platform` 在运行时判断
- 鸿蒙平台**不支持** `app-plus` 相关配置，涉及鸿蒙的特定需求禁止生成 `app-plus` 内容

#### 组件与样式

- **优先**使用 `uni_modules` 下的插件和 `uni-ui` 组件
- 样式优先使用 `rpx` 作为响应式单位
- 遵循 easycom 组件规范，组件放置在 `components/组件名/组件名.vue` 目录下，无需手动 import

### 服务端开发规范

#### 云对象设计

- 采用云对象作为主要开发模式，按业务模块划分（如 `user.co`、`order.co`）
- 方法命名采用「动词 + 名词」形式（如 `addUser`、`getOrderList`）
- 每个方法必须做好完整的错误处理，并返回标准化 JSON 格式：`{ errCode, errMsg, data }`

#### 数据库操作 (JQL)

- **必须**使用 JQL 语法操作数据库，**禁止**在云函数/云对象中拼接原生 SQL 语句
- 数据库表需设计对应的 `schema.json`，合理设置字段类型、必填校验及默认值
- **必须**通过 `db.schema` 配置表和字段的权限控制，**禁止**在客户端绕过权限直接操作数据库
- 对于高频查询字段，需在 `schema.json` 中配置索引以优化性能

#### 安全与权限

- 除公开接口（如登录、注册）外，云对象的每个方法**必须**校验用户身份（通过 `this.getUniIdToken()`）
- 校验输入参数的合法性，防止恶意注入
- 敏感操作必须进行严格的权限校验，避免越权操作

### 平台与工程限制

#### 插件使用限制

- UTS 插件**必须**放在 `uni_modules` 目录下，作为插件出现
- App 原生语言插件已不再维护，如需扩展原生能力，**必须**生成 UTS 插件代码
- **禁止**生成依赖 HBuilderX 特定版本或无法跨端使用的闭源付费插件代码

#### 鸿蒙开发特殊限制

- 鸿蒙平台和 App（安卓/iOS）是分开并立的，App 相关的配置和代码**不能**直接用在鸿蒙上
- 涉及鸿蒙权限配置时，需在 `harmony-configs` 目录下按照原生路径放置配置文件，**禁止**随意修改目录结构

#### 上架与审核规避

- 避免生成高度同质化的模板代码，以免触发 App Store 的 4.3(a) 拒绝，业务逻辑和 UI 需具备独特性
- 如涉及隐私权限（定位、相机），必须在代码中包含权限说明及合规的授权处理逻辑

### 目录结构与路径

- 遵循 uni-app 标准目录结构：`pages/`（页面）、`components/`（组件）、`{uniCloud目录名}/cloudfunctions/`（云函数/云对象）、`{uniCloud目录名}/database/`（数据表 Schema）
- 静态资源**必须**放置在 `static/` 目录下
- 云函数/云对象的实际部署目录名**禁止硬编码**为具体值（如 `uniCloud-aliyun/`），必须从 `overview.md` `## HBuilderX 建项方式` 中提取 `uniCloud 目录名` 字段，用该值替换所有 `{uniCloud目录名}` 占位符
- 如果 `overview.md` 未标注 `uniCloud 目录名`，视为设计缺失，先回退到 `uniapp-solution-architect` 补充

### 注释与代码规范

- 代码需具备清晰、规范的注释，关键业务逻辑和云对象方法必须说明用途、参数及返回值
- **禁止**生成冗余的 `console.log` 调试代码
- 云函数/云对象日志中**禁止**打印敏感信息（密码、Token、完整手机号）
- 提交代码前清理所有调试用输出

### 规范遵守优先级

当上述规范与设计文档、任务卡中的特定约定冲突时，按以下优先级处理：

1. 设计文档 `docs/design/*.md` 中明确写明的特定约定（最高优先级）
2. 本节的代码生成强制规范
3. 任务卡中的执行边界说明

如果设计文档的约定与本规范冲突且无法确定是否有意偏离，先暂停并回退到 `uniapp-solution-architect` 确认。

## `schema2code` 默认工作流

当当前任务具备以下条件时，优先判断是否进入 `schema2code` 流程：
- 用户在做新业务模块
- 需要同时落数据、云端接口、前端页面或后台页面
- 当前项目已采用 `uniCloud` 作为主数据与服务承载方式

默认顺序：

1. 数据结构
2. 云对象或云函数
3. 前端页面
4. 管理后台页面

默认输出关注点：
- 数据结构：是否需要新增或修改 `schema`
- 云端：先判断 `clientDB + DB Schema + uni-id` 是否覆盖，再决定是否需要云对象/云函数
- 前端：是否需要列表页、表单页、详情页
- 后台：是否需要 `uni-admin` 页面和菜单接入

如果当前任务只覆盖其中一段，例如“只补 schema”或“只补前端页面”，则只执行对应段，不重定义整条链路。

如果流程中引入了非官方初始化的数据表、集合或字段，本轮必须同步更新 `docs/design/data-model.md`。

如果流程中形成了明确调用链或状态流转，本轮必须同步更新 `docs/design/flows.md`。

## 前端执行规则

做前端时，不要把功能修补、UI 调整、交互补丁分成三轮。

每轮前端开发都同时检查：
- 当前页面属于哪种页面模式
- 当前页面的统一主流程是否已经在各目标端保持一致
- 是否能直接复用官方模块或项目内已有官方封装
- 是否只是样式统一问题，而不是结构重写问题
- Web 横屏与移动竖屏需要做到什么层级的一致

默认规则：
- 业务操作页：移动优先，Web 增强
- 内容/展示/营销页：移动竖屏与 Web 横屏一起考虑
- 通用页优先官方模块改造
- 组件少但整体表现要求高的页面，允许整体实现，但仍优先复用官方基础件

风格统一默认遵循 `ui-ux-pro-max`，但不重做设计阶段已经确定的风格边界。

需要查表时，阅读：
- [前端页面模式](references/frontend-page-patterns.md)

## 平台差异执行规则

遇到平台问题时，先分类：

1. 编译期差异
2. 运行期差异
3. 能力差异与降级

处理顺序：

1. 先确认统一主流程是否已经共享
2. 再确认当前步骤是否真的需要平台专门处理
3. 能共享实现时不提前拆分
4. 能降级、替代、隐藏时优先这样做
5. 必要时再隔离平台专门实现

Harmony、小程序、Web、App 都先按同一产品主线实现。只有出现明确发行要求或能力差异时，才记录对应技术隔离点；小程序不默认要求能力完全一致。

不要因为某端存在能力限制，就把这个端直接实现成另一种产品角色。应先保住统一业务主线，再对受限能力做裁剪、替代或隐藏。

如果上游设计没有明确发行端，不要阻塞开发，也不要自行拆成多套业务。按 uni-app 多端兼容主线实现，并把实际遇到的条件编译、运行期判断、能力降级记录回设计文档。

如果当前任务涉及公开 Web 页面、SEO、营销落地页、可分享内容页，再确认设计是否已经判断：
- 是否需要 SSR
- 是否使用前端网页托管
- 是否需要 `uni-ssr` 云函数
- Web、SSR 云函数、业务云函数是否同服务空间
- 跨域、cookie、token、请求域名如何处理

如果设计把 H5/Web 写成“轻端”或把某个发行端写成另一个产品角色，先回退补设计。SSR/托管只在产品目标需要公开 Web 能力时作为技术分支处理。

如果当前任务涉及多个应用共用同一个 `uniCloud` 服务空间，执行前至少核验：
- 当前官方库是否真的支持多应用共享
- 当前集合、云对象、云函数是否已经具备应用边界
- 当前云存储路径是否按应用做命名空间隔离
- 当前凭据、支付、内容安全、消息配置是否逐应用分离
- 当前实现是否错误地把 `dcloudAppid` 当成唯一授权边界
- 当前文件上传、临时链接、删除动作是否仍由云端统一控权

需要查表时，阅读：
- [平台差异核对清单](references/platform-diff-checklist.md)

## 部署与可观测性

### uniCloud 服务空间环境

项目必须明确三套服务空间环境的用途：

| 环境 | 用途 | 数据特征 |
|------|------|---------|
| **开发空间** | 本地 HBuilderX 联调、daily build | 独立数据，可随意清理 |
| **测试/预发布空间** | QA 验收、回归测试 | 需定期用生产数据脱敏后同步 |
| **生产空间** | 正式发行 | 真实用户数据，严格权限 |

**强制规则**：
- `manifest.json` 和 `package.json` 中的服务空间配置不得硬编码，必须通过环境变量或 `.env.*` 文件区分
- 切换服务空间后必须验证：云函数连通性（ping）、clientDB 权限、文件存储路径、支付回调地址
- 不同 `dcloudAppid` 对应不同服务空间，禁止跨空间混用

### 发行与版本管理

**HBuilderX 发行前检查清单**：
1. `manifest.json` 基础配置完整（AppID、版本号、版本名称）
2. `pages.json` tabBar 路径与实际页面文件一一对应
3. 条件编译标识（`#ifdef`、`#ifndef`）覆盖所有平台差异代码段
4. 云函数 `package.json` 无 `^`/`~` 版本锁定，所有依赖已上传
5. DB Schema 文件已在目标服务空间导入并校验通过
6. 隐私协议提示（`uni-app` 强制要求）已配置

**版本号规范**：
- 主版本.次版本.修订号（`1.0.0`）
- 主版本：破坏性变更（如数据结构不兼容、API 路径重构）
- 次版本：新增功能（向后兼容）
- 修订号：Bug 修复、小优化
- 每次正式发行后同步更新 `docs/tracking/change-log.md`

**灰度发布**：
- uniCloud 不支持原生灰度，依赖前端条件编译或运行期特性开关
- 高风险操作（支付、批量删除、权限变更）必须通过 `uniCloud-async` 或任务队列削峰
- 发行包留存最近 3 个版本（方便回滚）

### 可观测性要求

以下指标必须在交付前确认接入：

**云函数监控**：
- 开启 uniCloud 控制台的云函数日志（所有云函数默认记录调用次数、耗时、错误）
- 关键业务云函数（转写提交、支付回调、权益发放）设置超时告警阈值（如 > 30s）
- 长耗时任务（转写、批量导出）必须实现进度回调或任务状态查询接口，禁止静默超时

**前端埋点与错误监控**：
- 推荐接入 `uni-stat`（DCloud 官方统计）：页面 PV/UV、留存、崩溃率
- 生产环境 H5/App 必须接入错误监控（如 Sentry），至少覆盖：
  - 未捕获的 JS 异常
  - 云函数调用失败的业务错误码
  - 云存储上传失败的错误日志
- 关键业务节点（任务提交、支付发起、会员购买）记录行为埋点

**日志规范**：
- 云函数日志使用 `console.log`/`console.error`，禁止打印敏感信息（密码、Token、完整手机号）
- 错误日志必须包含：错误码、用户标识（uid）、请求 ID（用于链路追踪）、时间戳
- 运维/审计类操作（管理员导出、放行风控、修改会员资格）必须写入 `rc_audit_log`，字段覆盖：operator_id、target_id、operation_type、before_snapshot、after_snapshot、操作时间

**验收门槛**：
- 部署完成前必须确认云函数可调用、数据可写入、监控看板可查看
- 在 `docs/tracking/change-log.md` 中记录部署时间、服务空间版本、部署人

## 不要这样做

不要：
- 在范围不清时直接写代码
- 因为赶进度跳过官方复用检查
- 没判断 `clientDB` / `DB Schema` 就直接写云函数 CRUD
- 没有 URL 化鉴权与幂等设计就接第三方回调
- 没有生命周期规则就实现云存储文件上传
- 把最终发行端选择当成开发阶段的产品分叉理由
- 没有产品目标依据就把 H5/Web 当成裁剪版端
- 因为用户或上游设计已经说"可以共享"，就跳过共享风险核验
- 把 `dcloudAppid` 当成认证、授权、支付、存储、凭据隔离的唯一手段
- 把当前步骤扩成大面积重构
- 把平台差异塞进临时补丁却不回写
- 代码完成后不更新文档
- 接受 `columns: []` 格式的数据模型——如果 data-model.md 使用了非官方格式，执行前必须修正为 `properties: {}` 格式
- 任务卡目标目录名与 overview.md 建项结论不一致——如果发现不一致，先修正任务卡再开始实施
- 代码或数据变更后不更新 `docs/project-system.md` 的对应状态
