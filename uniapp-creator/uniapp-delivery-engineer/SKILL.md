# uniapp-delivery-engineer

## 文档信息

- 当前版本：`v1.2`
- 更新时间：`2026-05-10`
- 当前定位：面向执行阶段的官方优先交付 skill

## 概览

用于在既有设计边界内完成 uni-app/uniCloud 开发执行。重点不是“尽快写代码”，而是按当前步骤推进、优先复用官方能力、处理必要的平台差异，并保持文档可持续交接。

这个 skill 面向真正的开发落地：前端页面、uniCloud、后台协同、文档回写，都要在同一轮里闭环。

## 使用前提

只有在下面条件成立时才使用：
- 已经有明确的设计边界
- 已经知道本轮要改哪个模块编号、仓库或目录
- 已经明确官方模板基线或当前项目基线
- 已经能找到对应任务卡或实际执行步骤

如果范围、模板、拓扑、平台策略仍不清楚，先退回 `uniapp-solution-architect`。

## 开始前先读什么

改代码前按这个顺序读：

1. 先确认任务卡路径是否已经在设计输出中写明
2. 如果任务卡路径不明确，先读 `docs/project-system.md`
3. 当前任务卡或当前步骤
4. 相关设计文档
5. 如涉及数据，读 `docs/design/data-model.md`
6. 如涉及流程、状态、调用链、平台分支，读 `docs/design/flows.md`

不要只凭之前对话记忆直接开工。

## 默认执行顺序

按下面顺序做：

1. 确认本轮只覆盖哪个模块编号和步骤
2. 确认涉及哪个仓库、目录和模块
3. 检查当前项目已有的官方封装和官方模块
4. 选择最小可行改动
5. 完成实现
6. 回写任务卡、总控、变更记录，以及必要的数据文档和流程文档

如果用户一次提了很多事，先把当前模块和当前步骤收窄，不要一轮混做整项目。

## 与 `writing-plans` / `executing-plans` / `test-driven-development` 的自然配合

执行时按任务类型自然选择：

- 结构或模块拆分不清：先 `writing-plans`
- 已有清晰步骤，进入实际落地：再 `executing-plans`
- 行为规则、状态流转、跨端差异、云端逻辑容易回归：优先 `test-driven-development`

不要把这些 skill 机械写进文档模板；只要让每一步天然能接上它们即可。

## 官方优先执行规则

固定优先顺序：

1. 项目内已有的官方封装
2. 官方模板、官方组件、官方插件、`uni_modules`、官方云能力
3. 最小改造
4. 薄业务包装
5. 官方路径确实不适合时，再做自定义实现

需要查表时，优先阅读：
- [模板选择指引](references/template-selection-guide.md)
- [官方生态核对清单](references/official-ecosystem-checklist.md)

## `schema2code` 默认工作流

当当前任务具备以下条件时，优先判断是否进入 `schema2code` 流程：
- 用户在做新业务模块
- 需要同时落数据、云端接口、前端页面或后台页面
- 当前项目已采用 `uniCloud` 作为主数据与服务承载方式

默认顺序：

1. 数据结构
2. 云对象或云函数
3. 前端页面
4. 管理后台页面

默认输出关注点：
- 数据结构：是否需要新增或修改 `schema`
- 云端：是否需要标准 CRUD、权限校验、参数校验
- 前端：是否需要列表页、表单页、详情页
- 后台：是否需要 `uni-admin` 页面和菜单接入

如果当前任务只覆盖其中一段，例如“只补 schema”或“只补前端页面”，则只执行对应段，不重定义整条链路。

如果流程中引入了非官方初始化的数据表、集合或字段，本轮必须同步更新 `docs/design/data-model.md`。

如果流程中形成了明确调用链或状态流转，本轮必须同步更新 `docs/design/flows.md`。

## 前端执行规则

做前端时，不要把功能修补、UI 调整、交互补丁分成三轮。

每轮前端开发都同时检查：
- 当前页面属于哪种页面模式
- 是否能直接复用官方模块或项目内已有官方封装
- 是否只是样式统一问题，而不是结构重写问题
- Web 横屏与移动竖屏需要做到什么层级的一致

默认规则：
- 业务操作页：移动优先，Web 增强
- 内容/展示/营销页：移动竖屏与 Web 横屏一起考虑
- 通用页优先官方模块改造
- 组件少但整体表现要求高的页面，允许整体实现，但仍优先复用官方基础件

风格统一默认遵循 `ui-ux-pro-max`，但不重做设计阶段已经确定的风格边界。

需要查表时，阅读：
- [前端页面模式](references/frontend-page-patterns.md)

## 平台差异执行规则

遇到平台问题时，先分类：

1. 编译期差异
2. 运行期差异
3. 能力差异与降级

处理顺序：

1. 先确认当前步骤是否真的需要平台专门处理
2. 能共享实现时不提前拆分
3. 能降级、替代、隐藏时优先这样做
4. 必要时再隔离平台专门实现

Harmony 不当作普通 App 分支处理，小程序不默认要求能力完全一致。

需要查表时，阅读：
- [平台差异核对清单](references/platform-diff-checklist.md)

## 数据与流程回写规则

如果执行中新增或修改了非官方初始化的数据表、集合或字段，本轮必须同步更新数据文档。

如果执行中新增或改变了：
- 业务流程
- 状态流转
- uniCloud 调用链
- 平台分支

本轮必须同步补 Mermaid。

不要把这些变化只留在代码里。

固定回写路径：
- 数据文档：`docs/design/data-model.md`
- 流程文档：`docs/design/flows.md`

## 收尾时必须回写什么

每轮有成果的修改，至少回写：

1. 任务卡执行记录与步骤状态
2. `docs/project-system.md` 的当前状态和下一步
3. `docs/tracking/change-log.md`
4. 如有数据变化，更新数据文档
5. 如有流程变化，更新流程文档

所有说明都要写在文档头部或执行记录中，明确：
- 修改成果
- 影响文件
- 当前版本号

需要回写顺序时，阅读：
- [文档工作流](references/documentation-workflow.md)

## 何时停止并回退到设计

出现以下情况，暂停执行并回到 `uniapp-solution-architect`：
- 本轮步骤跨越了原设计边界
- 需要改模板基线
- 发现共享后台或多应用边界与原方案不一致
- 需要新增明显的平台架构分支
- 官方路径已无法覆盖，但自定义方案会影响整体结构

## 不要这样做

不要：
- 在范围不清时直接写代码
- 因为赶进度跳过官方复用检查
- 把当前步骤扩成大面积重构
- 把平台差异塞进临时补丁却不回写
- 代码完成后不更新文档
