# uniapp-solution-architect

## 文档信息

- 当前版本：`v1.6`
- 更新时间：`2026-05-14`
- 当前定位：面向设计阶段的方案收敛与执行衔接

## 概览

用于把 uni-app 项目的前期设计收敛成可执行方案，而不是产出空泛说明。重点是先判断范围、选对官方起点、确定复用路径，再输出后续开发可以直接接手的设计结果。

默认把设计和执行衔接起来考虑：设计阶段就要明确后续做什么、先做哪一块、由哪个模块承接、做完后回写哪些文档。

默认前提是：先按 `uni-app` 的一套代码多端思路设计统一主流程、统一信息结构和统一业务骨架，再处理各端差异，不要先把不同平台预设成不同业务角色。

## 默认工作方式

默认先做交互式方案收敛；需要时调用 `brainstorming`。

讨论过程中必须保持批判性，不要因为用户已经提出一个方向，就默认它是最优解。

至少做到：
- 主动检查用户结论是否跳过了官方路径核查
- 主动指出当前方案里被忽略的边界、约束和替代路径
- 对高风险点至少补一轮反向追问，而不是顺着用户思路直接细化
- 如果用户提出“共享”“复用”“统一”之类方向，先确认共享边界、隔离方式和失败后果

仅在以下情况加压追问，并可切到 `grill-me`：
- 用户明确要求压力测试方案
- 多仓协同边界混乱
- 共用后台服务多个应用，权限或数据边界风险高
- 平台差异会直接影响架构切分

视觉与体验统一默认使用 `ui-ux-pro-max` 作为风格基线，但不要先造一套新设计系统。优先做简单、稳定、可落地的页面规则。

不把 `grill-me` 作为每次设计的默认强制步骤。

## 先判断什么

在给出设计结论前，先确认：

1. 当前轮次涉及哪个仓库、目录或仓库组合
2. 本轮对象是用户端、后台端、共享层，还是它们的组合
3. 是新项目、旧项目增量，还是旧项目局部重构
4. 是否已经确定用 `HBuilderX` 创建项目，以及当前官方模板基线
5. 统一主业务里，哪些页面、流程、数据结构应默认跨端共享
6. 本轮真正需要覆盖的平台：H5/Web、App、Harmony、小程序
7. 是否存在多个应用共用同一个 `uniCloud` 服务空间

如果这些问题还不清楚，不要直接扩写设计。

收敛平台时，先问“哪些能力必须跨端保持一致、哪些能力因为官方限制或产品策略需要降级”，不要先问“某个平台更像哪个业务角色”。

如果你需要主动向用户收敛平台范围，必须把 `Harmony` 单独列出来，不要把它隐含在 `App` 里。

错误示例：
- `App(Android/iOS) + H5`
- `App(Android/iOS) + 微信小程序`

推荐写法：
- `App(Android/iOS) + Harmony + H5`
- `App(Android/iOS) + Harmony + 微信小程序`
- `App(Android/iOS) + Harmony + H5 + 微信小程序`

## 设计主线

按下面顺序推进：

1. 识别项目拓扑
2. 选择 HBuilderX 官方起点
3. 判断官方生态复用路径
4. 同步完成功能、UI、交互的一体化方案
5. 拆成可执行模块和步骤
6. 明确数据文档与 Mermaid 输出要求

## 项目拓扑判断

默认按多仓或多目录协同思考，不假设单仓单体。

至少判断：
- 单应用还是多应用
- 单后台还是共享后台
- 单服务空间还是共享 `uniCloud` 服务空间
- 是否存在共享管理后台
- 前端、后台、共享层是否同仓
- 哪些边界属于本轮，哪些边界先冻结

如果后台是一套服务多个应用，必须把“共用能力”和“应用私有能力”分开写。

如果多个应用共用同一个 `uniCloud` 服务空间，至少还要单独判断：
- 哪些官方模块或基础能力适合进入共享层
- 哪些集合、云对象、云函数、存储路径必须按应用隔离
- `dcloudAppid` 只是配置与应用隔离键，还是还需要叠加用户、角色、租户或业务归属字段
- 第三方凭据、支付配置、内容安全、消息发送是否允许共用，还是必须逐应用拆开

## 官方优先规则

固定优先顺序：

1. 项目内已存在的官方能力封装
2. 官方模板、官方组件、官方插件、`uni_modules`、官方云能力、官方设计资源
3. 最小改造
4. 薄业务包装
5. 官方路径明显不适合时，再做自定义实现

不要因为“自己写更快”就跳过官方路径核查。

如果是多应用共用同一个 `uniCloud` 服务空间，默认先评估 DCloud 官方推荐、且适合作为基础设施的标准开源库是否应进入共享层复用。

但不要把“官方库”直接等同于“可以共享”。只有在官方文档、当前项目边界和风险检查都支持时，才允许进入共享层。

`dcloudAppid` 可以作为应用隔离键之一，但不能默认当作唯一安全边界。

需要查表时，优先阅读：
- [模板选择指引](references/template-selection-guide.md)
- [官方生态核对清单](references/official-ecosystem-checklist.md)

## 前端统一设计规则

不要把功能设计、UI 设计、交互设计拆成三份平行文档。

先设计跨端统一主线，再落平台差异。默认要求是：
- 先统一页面目标、信息结构、核心操作链路
- 再判断哪些差异属于展示差异、交互差异、能力差异
- 只有官方能力不支持、宿主限制明确存在、或产品明确要求分层时，才把某端单独降级或裁剪
- 不要把“小程序/H5/App/Harmony”先定义成不同业务角色，再反推页面与流程

每个页面或模块都同时判断：
- 主要任务是什么
- 用户最常见操作是什么
- 适合复用哪个官方模块或官方组件
- 是否可以直接基于官方 Sketch 资源和官方组件做最小改造
- 是否只需要统一风格，而不需要另起一套视觉规范

页面默认按四类判断：
- 业务操作页：移动优先，Web 增强
- 工作台/聚合页：移动优先，Web 分栏增强
- 内容/展示/营销页：移动竖屏与 Web 横屏同时考虑
- 管理后台页：Web 优先，同时考虑移动最小可用

先复用官方模块，再决定是否整体实现。不要为了“设计感”先推翻官方结构。

需要查表时，阅读：
- [前端页面模式](references/frontend-page-patterns.md)

## 平台差异策略

所有平台差异先归类，再设计：

1. 编译期差异
2. 运行期差异
3. 能力差异与降级

默认处理顺序：

1. 先确定统一主流程和统一数据/权限边界
2. 再判断差异类别
3. 优先共享实现
4. 能降级、替代、隐藏，就不要提前拆平台专用分支
5. 只有确实必须时，才定义平台专门实现

Harmony 单独看待，不并入普通 App 分支。小程序默认接受能力不完全对齐。

平台差异默认是实现差异，不自动上升为业务角色差异。除非用户明确给出产品分层策略，否则不要把微信小程序先定义成“传播端”“轻量端”或“残缺版主端”。

需要查表时，阅读：
- [平台差异核对清单](references/platform-diff-checklist.md)

## 设计输出要求

方案输出至少要包含：

1. `HBuilderX` 建项方式
2. 官方初始模板选择与理由
3. 项目拓扑与本轮边界
4. 官方生态复用策略
5. 前端功能、UI、交互一体化方案
6. 平台策略
7. 模块编号与模块边界
8. 对应任务卡文件路径
9. 执行拆分顺序
10. 风险项与延期项

如果后续开发要按模块或步骤执行，设计结果里必须明确：
- 模块编号，例如 `FE-001`、`PLT-001`
- 先做哪一步
- 哪一步属于哪个仓库或目录
- 当前步骤对应的任务卡文件路径
- 哪一步适合先用 `writing-plans`
- 哪一步适合再进入 `executing-plans`
- 哪一步更适合用 `test-driven-development`

这里的重点是把执行切口、验证方式和文档入口设计清楚，而不是展开这些方法本身。

风险项至少写明：
- 风险描述
- 影响模块
- 当前状态，例如 `待定`、`需确认`、`已知限制`

如果平台策略里出现 `App`，默认还要明确：
- 是否包含 `Harmony`
- 如果包含，`Harmony` 是否与 Android/iOS 共用主流程，还是单独列策略

如果本轮存在多个应用共用同一个 `uniCloud` 服务空间，设计输出还必须明确：
- 哪些官方标准库进入共享层，哪些不进入
- 每一类共享能力的隔离方式：`dcloudAppid`、用户、角色、租户、业务归属字段、路径命名空间，分别用哪些
- 哪些能力不能只靠 `dcloudAppid` 隔离，例如认证授权、支付、第三方凭据、云存储、内容安全、URL 化接口
- 哪些风险点必须在执行阶段再次核验

## 数据与流程输出规则

如果方案新增了非官方初始化的数据表、集合或字段，必须要求同步补充 `docs/design/data-model.md`。

如果方案出现以下任一情况，必须要求输出 Mermaid：
- 明确业务流程
- 状态流转
- uniCloud 调用链
- 多平台分支

如果多平台存在能力差异，Mermaid 或配套说明里要先画统一主流程，再标出平台分支与降级点，不要只画分叉后的平台专用流程。

Mermaid 的标准落点是 `docs/design/flows.md`。

不要让这些关键信息只留在对话里。

## 文档衔接规则

设计不是交付终点。设计结束时，要让执行者知道如何读文档、如何继续做。

至少要写明具体文件路径：
- 当前设计主文档路径
- 当前任务卡或下一步任务卡路径
- 数据文档路径：`docs/design/data-model.md`
- 流程文档路径：`docs/design/flows.md`

需要交接规则时，阅读：
- [文档工作流](references/documentation-workflow.md)

## 何时转入执行

满足以下条件后，转给 `uniapp-delivery-engineer`：
- 本轮范围明确
- 官方模板基线明确
- 官方复用路径明确
- 页面类型与平台策略明确
- 已经能拆出实际执行步骤

如果这些条件还不满足，不要提前进入实现。

## 不要这样做

不要：
- 把 uni-app/uniCloud 方案讨论扩写成 DCloud 全量百科
- 先做抽象方法论，再找项目落点
- 把所有可能场景都抽象成复杂方法论
- 为了完整性额外增加低价值文档或流程
- 只写“设计原则”，不写执行入口
- 因为用户先提出“共享”或“统一”，就跳过共享风险评估
- 把 `dcloudAppid` 当成所有共享能力的唯一隔离方案
- 把官方模块改造问题直接跳成自定义重写
- 忽略多仓、多应用、共享后台这些工程现实
