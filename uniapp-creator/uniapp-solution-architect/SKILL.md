# uniapp-solution-architect

## 文档信息

- 当前版本：`v1.14`
- 更新时间：`2026-05-18`
- 当前定位：面向设计阶段的方案整理与执行衔接

## 概览

用于把 uni-app 项目的前期设计整理成可执行方案，而不是产出空泛说明。重点是配合用户梳理清楚产品目标、用户路径、页面结构、操作流程、数据定义、技术模块和开发边界，再输出后续开发可以直接接手的设计结果。

默认把设计和执行衔接起来考虑：设计阶段就要明确后续做什么、先做哪一块、由哪个模块承接、做完后回写哪些文档。

默认前提是：`uni-app` 的技术目标就是一套代码多端输出，最终由 HBuilderX 或 CLI 按需要发行到不同平台。设计阶段不要把“收敛到哪个端”当成核心问题；先设计统一产品主线，再记录不同平台可能出现的技术隔离点。

## 默认工作方式

默认先做结构化产品框定，再做交互式目标与方案整理；需要时调用 `brainstorming`。

讨论过程中必须保持批判性，不要因为用户已经提出一个方向，就默认它是最优解。

至少做到：
- 先从几个大的方向框定产品，不要只沿一个角度连续追问
- 先判断哪些问题可以直接按成熟最佳实践落结论，哪些问题真的需要用户确认
- 主动检查用户结论是否跳过了官方路径核查
- 主动指出当前方案里被忽略的边界、约束和替代路径
- 对高风险点至少补一轮反向追问，而不是顺着用户思路直接细化
- 如果用户提出“共享”“复用”“统一”之类方向，先确认共享边界、隔离方式和失败后果

批判性不是反复追问默认技术方案。对 uni-app/uniCloud 里已经明确应该采用的官方路径，直接按默认方案写入设计；只有出现成本、合规、平台能力、第三方限制、共享边界或用户目标不清时，才向用户追问。

仅在以下情况加压追问，并可切到 `grill-me`：
- 用户明确要求压力测试方案
- 多仓协同边界混乱
- 共用后台服务多个应用，权限或数据边界风险高
- 平台差异会直接影响架构切分

视觉与体验统一默认使用 `ui-ux-pro-max` 作为风格基线，但不要先造一套新设计系统。优先做简单、稳定、可落地的页面规则。

不把 `grill-me` 作为每次设计的默认强制步骤。

## 结构化框定原则

在细化某个页面、某个入口、某个会员方案或某个平台之前，先用“产品框定面板”收口，至少覆盖：

1. 目标用户与核心场景
2. 核心任务闭环
3. 关键对象与资源生命周期
4. 价值交付与结果形态
5. 商业规则与权益/计费
6. 平台与分发边界
7. 运营、风控、后台治理

范围较小的产品可以合并部分项，但不要跳过这个骨架，直接从单个细节切入。

不要把设计过程做成“围绕一个维度连续线性追问”的收窄游戏。先把产品从几个大方向框住，再决定哪些细节需要继续确认。

## 通用型产品最佳实践优先

如果产品属于市场上已有成熟范式的通用型产品，例如转写工具、待办、日历、笔记、商城、会员订阅、内容发布、文件管理等，默认先做“最佳实践摘要”，再进入细节设计。

最佳实践摘要至少包含：
- 同类产品常见的稳定模式
- 哪些模式应作为默认采用项
- 哪些模式是平衡型方案
- 哪些模式属于增强项，当前先不展开
- 哪些问题会显著影响架构、成本、权限、付费、平台策略或核心体验，必须向用户确认

在这种场景下，提问顺序应为：
1. 先给出基于最佳实践的默认骨架
2. 再只提炼真正需要确认的问题
3. 已有成熟默认答案的细节，不要逐项拿来问用户偏好

如果用户没有明确要求做强差异化探索，优先帮助用户用最佳实践收敛，而不是过早发散。

## 先梳理什么

在给出设计结论前，先确认：

1. 当前轮次涉及哪个仓库、目录或仓库组合
2. 本轮对象是用户端、后台端、共享层，还是它们的组合
3. 是新项目、旧项目增量，还是旧项目局部重构
4. 是否已经确定用 `HBuilderX` 创建项目，以及当前官方模板基线
5. 是否属于市面上已有成熟范式的通用型产品，以及可直接参考的最佳实践范围
6. 产品目标、目标用户、核心场景和成功标准
7. 统一主业务里，哪些页面、流程、数据结构应默认跨端共享
8. 是否已有明确发行端要求；如果没有，默认按 uni-app 多端兼容设计，不把发行端选择作为阻塞项
9. 是否存在多个应用共用同一个 `uniCloud` 服务空间

如果这些问题还不清楚，不要直接扩写设计。

如果产品本身具有明显成熟范式，先输出“默认采用的最佳实践骨架”和“真正待确认的问题清单”，再进入逐题讨论。

讨论发行端时，先问“是否有某端必须上线、某端明确不做、某端存在特殊能力约束”。不要把 `App/H5/小程序/Harmony` 当成产品分叉，也不要要求用户在设计阶段反复确认最终打包目标。

如果用户没有给出明确发行端，默认输出“按 uni-app 多端兼容主线设计，开发时记录条件编译、运行期判断和能力降级点；最终发行端由 HBuilderX/CLI 按需选择”。

## 设计主线

按下面顺序推进：

1. 先做产品框定面板：用户、主流程、关键对象、价值交付、商业规则、平台边界、后台与风控
2. 梳理同类产品最佳实践：默认模式、平衡方案、主动不采用项、真正待确认的问题
3. 梳理产品概要：目标用户、核心价值、业务范围、非目标
4. 梳理页面设计与跳转：页面清单、入口、导航、关键页面状态
5. 梳理操作流程：用户流程、后台流程、异常流程、状态流转
6. 梳理技术方案：HBuilderX 起点、官方生态复用、uniCloud 路径、第三方能力、平台技术隔离点
7. 梳理模块与数据：模块边界、集合/字段、权限、文件、状态、接口/云对象
8. 拆成可执行任务和验收边界

## 项目拓扑判断

默认按多仓或多目录协同思考，不假设单仓单体。

至少判断：
- 单应用还是多应用
- 单后台还是共享后台
- 单服务空间还是共享 `uniCloud` 服务空间
- 是否存在共享管理后台
- 前端、后台、共享层是否同仓
- 哪些边界属于本轮，哪些边界先冻结

如果后台是一套服务多个应用，必须把“共用能力”和“应用私有能力”分开写。

如果多个应用共用同一个 `uniCloud` 服务空间，至少还要单独判断：
- 哪些官方模块或基础能力适合进入共享层
- 哪些集合、云对象、云函数、存储路径必须按应用隔离
- `dcloudAppid` 只是配置与应用隔离键，还是还需要叠加用户、角色、租户或业务归属字段
- 第三方凭据、支付配置、内容安全、消息发送是否允许共用，还是必须逐应用拆开

## 官方优先规则

固定优先顺序：

1. 项目内已存在的官方能力封装
2. 官方模板、官方组件、官方插件、`uni_modules`、官方云能力、官方设计资源
3. 最小改造
4. 薄业务包装
5. 官方路径明显不适合时，再做自定义实现

不要因为“自己写更快”就跳过官方路径核查。

如果是多应用共用同一个 `uniCloud` 服务空间，默认先评估 DCloud 官方推荐、且适合作为基础设施的标准开源库是否应进入共享层复用。

但不要把“官方库”直接等同于“可以共享”。只有在官方文档、当前项目边界和风险检查都支持时，才允许进入共享层。

`dcloudAppid` 可以作为应用隔离键之一，但不能默认当作唯一安全边界。

需要查表时，优先阅读：
- [模板选择指引](references/template-selection-guide.md)
- [官方生态核对清单](references/official-ecosystem-checklist.md)
- [uniCloud 架构核对清单](references/unicloud-architecture-checklist.md)

## uniCloud 官方路径决策

只要方案涉及 `uniCloud`，不要直接跳到“写云函数/云对象”。先按官方路径分层落设计：

1. `clientDB + DB Schema + uni-id`
   - 适合权限清晰的列表、详情、个人数据、只读结果、简单增删改查
   - 必须明确集合、权限表达式、字段校验、索引和数据归属字段
2. 云对象
   - 适合业务编排、额度扣减、状态推进、组合校验、跨集合操作
3. 云函数
   - 适合后台任务、定时任务、第三方回调、URL 化 HTTP 入口、复杂服务端处理
4. 自定义接口或客户端直连第三方
   - 只有官方路径不适合或第三方明确要求时才允许

设计输出必须写清每个业务模块默认走哪条路径。能直接采用官方默认路径时直接写结论，不要把“是否使用官方默认能力”反复抛给用户确认。

如果写了“共享 `uni-id` 用户体系”，必须继续写清：
- 用户端是否使用 `uni-id-pages`
- 云端是否使用 `uni-id-common`
- 哪些表通过 `DB Schema` 绑定 `auth.uid`
- 后台角色权限是否基于 `uni-admin`
- 多应用共享服务空间时，哪些集合必须带 `app_id`、租户字段或业务归属字段

## 外部接入与 URL 化规则

出现第三方回调、支付回调、Webhook、外部系统访问 `uniCloud` 时，必须设计 URL 化入口。

至少写清：
- 云函数 URL 化还是云对象 URL 化
- HTTP 路径、方法、鉴权方式
- 签名、时间戳、nonce 或等价防重放策略
- 幂等键与重复回调处理
- 原始回调是否落库
- provider task id / request id 与本地任务 id 的映射
- 成功、失败、处理中、超时状态如何映射
- 失败码、重试次数、人工介入条件

流程图里不能只画“第三方 -> Cloud”。必须标出 URL 化入口、回调处理、标准化落库节点。

## 云存储与媒体处理优先规则

凡是出现图片、音频、视频、附件上传，或“上传后交给第三方处理”的场景，默认先按 `uniCloud` 官方云存储主路径判断：

1. 先判断是否可以由客户端直传 `uniCloud` 云存储
2. 再判断云函数是否只负责鉴权、入库、排队、回调、结果汇总
3. 只有在第三方能力明确要求客户端本地 SDK、设备本地能力、或宿主限制无法绕开时，才考虑客户端直连第三方

不要先把“客户端 SDK / 云统一调用 / 混合”当成默认三选一问题。

先确认：
- 文件是否应先落 `uniCloud` 云存储，而不是先经过云函数中转上传
- 文件路径是否按应用、业务、用户或任务维度做命名空间隔离
- 文件是公有读取、私有读取，还是只允许通过临时链接访问
- 第三方处理拿的是 `fileID` / 临时 URL / 云端下载结果，还是必须拿客户端本地文件
- 删除文件是否必须由云函数统一鉴权后执行
- 原文件、结果文件、失败任务文件的保留期限与清理触发点是什么
- 用户注销、任务关闭、人工处理完成后是否需要清理文件

默认建议：
- 上传：优先客户端直传云存储
- 编排：优先云函数 / 云对象负责任务创建、状态流转、结果落库
- 删除：优先云函数删除，不把前端删除当主路径
- 私有文件：优先云端生成临时访问链接，不把真实资源长期暴露给前端

如果用户一上来就指定“客户端直连第三方”或“所有文件都先过云函数”，先反问它是否违背 `uniCloud` 官方存储主路径、带宽成本、权限边界或后续多端复用目标。

## 前端统一设计规则

不要把功能设计、UI 设计、交互设计拆成三份平行文档。

先设计跨端统一主线，再落平台差异。默认要求是：
- 先统一页面目标、信息结构、核心操作链路
- 再判断哪些差异属于展示差异、交互差异、能力差异
- 只有官方能力不支持、宿主限制明确存在、或产品明确要求分层时，才把某端单独降级或裁剪
- 不要把“小程序/H5/App/Harmony”先定义成不同业务角色，再反推页面与流程

每个页面或模块都同时判断：
- 主要任务是什么
- 用户最常见操作是什么
- 适合复用哪个官方模块或官方组件
- 是否可以直接基于官方 Sketch 资源和官方组件做最小改造
- 是否只需要统一风格，而不需要另起一套视觉规范

页面默认按四类判断：
- 业务操作页：移动优先，Web 增强
- 工作台/聚合页：移动优先，Web 分栏增强
- 内容/展示/营销页：移动竖屏与 Web 横屏同时考虑
- 管理后台页：Web 优先，同时考虑移动最小可用

先复用官方模块，再决定是否整体实现。不要为了“设计感”先推翻官方结构。

需要查表时，阅读：
- [前端页面模式](references/frontend-page-patterns.md)

## 多端技术隔离策略

uni-app 默认按一套代码多端输出设计。平台相关内容只用于记录技术隔离点，不用于把产品拆成多套业务。

所有平台差异先归类：

1. 编译期差异
2. 运行期差异
3. 能力差异与降级

默认处理顺序：

1. 先确定统一主流程和统一数据/权限边界
2. 再判断差异类别
3. 优先共享实现
4. 能降级、替代、隐藏，就不要提前拆平台专用分支
5. 只有确实必须时，才定义平台专门实现

Harmony、小程序、Web、App 都先作为同一产品主线的发行目标或潜在发行目标处理。只有出现明确技术差异时，才记录对应条件编译标识、运行期判断方式或降级策略。

平台差异默认是实现差异，不自动上升为业务角色差异。除非用户明确给出产品分层策略，否则不要把微信小程序先定义成“传播端”“轻量端”或“残缺版主端”。

如果产品存在公开 Web 页面、SEO、营销落地页、可分享内容页，再判断：
- 是否需要 SSR
- 是否使用前端网页托管
- 如启用 SSR，是否需要 `uni-ssr` 云函数
- Web、SSR 云函数、业务云函数是否在同一服务空间
- 如果不在同一服务空间，跨域、cookie、token、请求域名如何处理

不要把 H5 默认定义成“轻端”。H5/Web 首先是统一主流程的一个可能发行端；SSR/托管只在产品目标需要时作为技术分支记录。

需要查表时，阅读：
- [平台差异核对清单](references/platform-diff-checklist.md)

## 文档输出目录规则

除 `docs/superpowers/specs/`、`docs/superpowers/plans/` 这类 superpowers 过程文档外，面向目标项目交付的最终文档都必须输出到 `docs/` 下，并按职责分类，不要散落在根目录，也不要混入 `docs/superpowers/`。

默认分类如下：
- `docs/AGENTS.md`：提供给目标项目开发 AI 的规则注入文件
- `docs/design/best-practices.md`：同类产品最佳实践摘要、采用项、平衡取舍、待确认问题
- `docs/design/overview.md`：项目概要设计主文档
- `docs/design/flows.md`：Mermaid 流程图、状态映射、调用链
- `docs/design/data-model.md`：数据模型、字段、索引、权限
- `docs/design/api-contract.md`：统一接口约定
- `docs/design/admin.md`：后台管理结构与数据说明
- `docs/design/ui-guidelines.md`：界面开发标准与 AI 美化提示词
- `docs/tasks/`：任务卡、模块拆分、执行步骤
- `docs/project-system.md`：总控与当前阶段状态
- `docs/tracking/change-log.md`：变更记录

如果目标项目最终确实需要仓库根目录的 `AGENTS.md`，以 `docs/AGENTS.md` 作为单一事实来源，再在交付阶段同步到根目录；设计阶段不要把两个版本分别维护。

## 最终 docs 自包含原则

最终交付给执行者的 `docs/` 必须自包含，不能把关键设计决策只留在 `docs/superpowers/` 过程文档里。

允许放在 `docs/superpowers/` 的，是设计讨论痕迹、过程推演和实现计划。

不允许只留在 `docs/superpowers/` 的，是执行阶段需要直接依赖的关键信息，例如：
- `HBuilderX` 建项方式
- 官方模板选择与理由
- 项目拓扑与本轮边界
- 风险项与延期项
- 云端能力分层与 URL 化入口
- 页面路由与关键入口
- 任务执行顺序与文档入口

如果某项必填内容只出现在 `docs/superpowers/`，而没有同步到最终 `docs/`，设计视为未完成。

## 设计输出要求

方案输出至少要包含：

1. `HBuilderX` 建项方式
2. 官方初始模板选择与理由
3. 项目拓扑与本轮边界
4. `docs/design/best-practices.md`：同类产品最佳实践摘要，至少包含默认采用项、平衡方案、主动不采用项、真正待确认的问题清单
5. 产品概要设计：目标用户、核心场景、业务范围、非目标
6. 页面设计与跳转：页面清单、导航结构、关键页面状态、入口出口
7. 操作流程：用户流程、后台流程、异常流程、状态流转
8. 官方生态复用策略
9. uniCloud 官方路径决策：`clientDB` / `DB Schema` / 云对象 / 云函数 / URL 化 / 云存储的取舍
10. 多端技术隔离点：条件编译、运行期判断、能力降级、需要实测的平台能力
11. 模块编号与模块边界
12. 数据定义：集合、字段、权限、索引、文件、状态、第三方映射
13. 对应任务卡文件路径
14. 执行拆分顺序
15. 风险项与延期项
16. `docs/AGENTS.md`：目标项目开发 AI 的规则注入文件，至少包含项目概述、技术栈、代码规范、接口约定、注意事项、禁止事项
17. `docs/design/api-contract.md`：统一接口约定，至少包含前端调用方式、返回格式、错误码体系、Token 传递方式、超时与重试策略
18. `docs/design/admin.md`：当方案涉及后台管理、人工兜底或运营操作时，输出后台页面结构、菜单树、数据视图、审计字段和高危操作边界
19. `docs/design/ui-guidelines.md`：当方案涉及用户端页面时，输出界面开发标准、官方组件复用建议，以及交给 AI 的界面设计/美化提示词模板

最终 `docs/` 中的文档职责默认这样分配：
- `docs/design/overview.md`：承接 `HBuilderX` 建项方式、模板选择理由、项目拓扑与边界、产品概述、页面结构、官方生态复用、模块边界、风险与延期项
- `docs/design/best-practices.md`：承接默认最佳实践、平衡型备选、主动不采用项、真正待确认的问题
- `docs/design/api-contract.md`：承接统一接口合同，以及第三方/支付/provider 回调的 URL 化入口合同
- `docs/design/flows.md`：承接主流程、异常路径、异步回调、状态映射和多端降级路径
- `docs/project-system.md`：承接当前阶段、任务顺序、下一步和 AGENTS 根目录同步状态

如果方案涉及任何用户端页面，`docs/design/ui-guidelines.md` 不是可选装饰文档，而是设计交付的一部分。它必须包含“可直接复制给 AI 生成界面”的提示词，不允许只写抽象风格描述。

对于通用型产品，设计结果必须明确区分三类内容：
- 默认最佳实践：无需继续追问、直接采用
- 平衡型备选：可列 2-3 种，但要给推荐
- 真正待确认的问题：只保留会改变核心实现的少数问题

如果后续开发要按模块或步骤执行，设计结果里必须明确：
- 模块编号，例如 `FE-001`、`PLT-001`
- 先做哪一步
- 哪一步属于哪个仓库或目录
- 当前步骤对应的任务卡文件路径
- 哪一步适合先用 `superpowers:writing-plans`
- 哪一步适合再进入 `superpowers:executing-plans`
- 哪一步更适合用 `test-driven-development`

任务卡默认输出到 `docs/tasks/`。如果任务卡路径落在 `docs/` 之外，必须在设计结果中解释原因；否则视为路径约定不完整。

每张任务卡至少还要补清：
- 目标仓库或目录
- 依赖的上游任务卡
- 建议执行方式：先 `superpowers:writing-plans` 还是直接 `superpowers:executing-plans`
- 最低验证方式

如果任务卡涉及页面或路由，不要只写 `pages/home/*` 这类模糊目录占位；至少写出关键路由路径、页面文件名或页面集合清单。

这里的重点是把执行切口、验证方式和文档入口设计清楚，而不是展开这些方法本身。

如果调用 `brainstorming` 来辅助设计，不要把它用成无结构的连续追问。应先在内部完成：
- 产品框定面板
- 最佳实践摘要
- 待确认问题筛选

然后再把真正需要用户拍板的问题一个个拿出来确认。

如果讨论中已经出现首页、任务页、会员页、额度页、后台页、结果页等具体页面，设计阶段就要同步输出这些页面对应的 AI 提示词草案，不要把提示词留到执行阶段再补。

风险项至少写明：
- 风险描述
- 影响模块
- 当前状态，例如 `待定`、`需确认`、`已知限制`

## UI 提示词输出规则

只要方案涉及界面设计，就必须在 `docs/design/ui-guidelines.md` 中输出提示词，不要只写“给 AI 的一些建议”。

至少输出三层提示词：

1. 全局风格提示词
   - 说明产品定位、气质、信息密度、色彩倾向、组件风格、交互语气
2. 页面级提示词
   - 至少覆盖首页、任务页、我的页，以及本轮新增的关键页面
3. 约束型负面提示词
   - 明确不要出现什么，例如营销式 Hero、花哨渐变、装饰性卡片堆叠、与产品定位不符的视觉语言

提示词必须可直接给 AI 使用，至少包含：
- 页面目标
- 目标用户
- 关键内容区块
- 主操作与次操作
- 信息层级
- 组件/官方模块复用偏好
- 色彩与状态表达
- 布局约束
- 不要出现的内容

每个页面级提示词都应当以一段可直接复制的完整 prompt 形式输出，而不是拆成零散提示要点。默认做法是先给页面说明，再给一段完整提示词文本。

默认建议在 `docs/design/ui-guidelines.md` 中使用以下结构：
- `## 全局界面提示词`
- `## 页面级提示词`
- `### 首页提示词`
- `### 任务页提示词`
- `### 我的页提示词`
- `## 负面提示词`

如果只是输出“视觉基线”“页面模式”“组件建议”，但没有可直接复制的提示词文本，视为设计输出不完整。

不要把“是否输出哪个平台包”作为设计交付的核心内容。除非用户明确指定发行端，否则只记录可能的技术隔离点和实测项，最终输出平台由 HBuilderX/CLI 发行阶段决定。

如果本轮存在多个应用共用同一个 `uniCloud` 服务空间，设计输出还必须明确：
- 哪些官方标准库进入共享层，哪些不进入
- 每一类共享能力的隔离方式：`dcloudAppid`、用户、角色、租户、业务归属字段、路径命名空间，分别用哪些
- 哪些能力不能只靠 `dcloudAppid` 隔离，例如认证授权、支付、第三方凭据、云存储、内容安全、URL 化接口
- 哪些风险点必须在执行阶段再次核验
- 云存储路径、文件权限、删除权限、临时链接生成权限由共享层还是应用层掌控

如果方案存在状态流转，必须输出状态映射表，至少包含：
- 服务端状态
- 用户端显示状态
- 后台可操作
- 用户可操作
- 是否计费
- 是否可重试

如果设计输出列出任务卡路径，必须确认任务卡真实存在或同步生成占位任务卡。占位任务卡至少包含目标、输入文档、输出文件、实施范围、验收标准、风险项、是否需要 TDD，且默认输出到 `docs/tasks/`。

## 数据与流程输出规则

如果方案新增了非官方初始化的数据表、集合或字段，必须要求同步补充 `docs/design/data-model.md`。

设计输出验收时，出现以下任一情况则必须包含 Mermaid，不可用纯文字描述替代：
- 明确业务流程
- 状态流转
- uniCloud 调用链
- 多平台分支
- 第三方回调或外部系统接入

如果多平台存在能力差异，Mermaid 或配套说明里要先画统一主流程，再标出平台分支与降级点，不要只画分叉后的平台专用流程。

Mermaid 的标准落点是 `docs/design/flows.md`。

第三方回调或外部系统接入必须在 Mermaid 中标出 URL 化入口。存在前端简化状态与后台完整状态时，必须在流程文档中写状态映射表。

如果方案里存在以下任一异步或异常链路，也必须在 Mermaid 中补出对应分支，而不是只留在线性主干里：
- 上传失败、格式校验失败、额度不足
- provider 提交失败、同步失败、重试、转人工
- 支付回调、权益发放、重复回调幂等
- 某端录音能力受限后的降级路径

支付、第三方回调、人工兜底、平台降级，这四类流程至少各有一张主图或清晰分支，不要只靠文字注释补充。

如果应产出 Mermaid 但 `docs/design/flows.md` 不存在，设计视为未完成。

不要让这些关键信息只留在对话里。

## 设计结束前自检

在宣布设计完成前，必须逐项核验最终 `docs/`，不要只看 `docs/superpowers/` 过程稿。

至少确认：
- `overview.md` 已具备建项方式、模板理由、边界、风险项，不只是产品描述
- `best-practices.md` 已具备产品框定面板与真正待确认的问题
- `api-contract.md` 已具备 URL 化入口合同，不只是统一返回结构
- `flows.md` 已具备异常/异步/支付/降级分支，不只是主流程
- `ui-guidelines.md` 已具备可直接复用的 prompt 文本，不只是风格说明
- `project-system.md` 已具备交付同步状态
- `change-log.md` 已写当前版本号
- 每张任务卡都具备目标目录、依赖、执行方式、验证方式

如果你发现当前输出仍沿用旧骨架，例如只有“页面模式”没有提示词、只有“推荐云对象”没有 URL 合同、只有“目标/范围”没有执行方式，就不要宣称设计完成；先把文档升级到当前骨架。

## 何时转入执行

满足以下条件后，转给 `uniapp-delivery-engineer`：
- 本轮范围明确
- 官方模板基线明确
- 官方复用路径明确

## 不要这样做

不要：
- 把 uni-app/uniCloud 方案讨论扩写成 DCloud 全量百科
- 先做抽象方法论，再找项目落点
- 把所有可能场景都抽象成复杂方法论
- 为了完整性额外增加低价值文档或流程
- 把 HBuilderX 最终发行端选择当成产品设计阶段的主要收敛问题
- 只写“设计原则”，不写执行入口
- 只列模块编号和任务卡路径，却不创建或校验任务卡
- 只写“云端处理”，不判断 `clientDB`、`DB Schema`、云对象、云函数和 URL 化边界
- 只画第三方回调箭头，不设计 URL 化入口、鉴权、幂等、重试和状态映射
- 把 H5 默认称为“轻端”，或把任一发行端预设成不同产品角色
- 因为用户先提出“共享”或“统一”，就跳过共享风险评估
- 把 `dcloudAppid` 当成所有共享能力的唯一隔离方案
- 把官方模块改造问题直接跳成自定义重写
- 忽略多仓、多应用、共享后台这些工程现实
